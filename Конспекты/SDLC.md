# SDLC – конспект (6 фаз + типичные проблемы и решения)

![[Pasted image 20260121112655.png]]

## 1. Сбор и анализ требований  
**Цель:** понять, какие проблемы нужно решить и какие цели ставит заказчик.  

- **Проблема:** неясные, часто меняющиеся требования; разрыв в понимании между заказчиком и командой.  
- **Решение:** сформировать чёткий, короткий документ с требованиями; использовать прототипы/макеты для уточнения; планировать только контролируемое.

## 2. Документирование требований  
**Цель:** зафиксировать всё в официальном SRS (Software Requirement Specification).  

- **Проблема:** многостраничный, перегруженный список требований.  
- **Решение:** выделять высокоуровневые требования, дополнять ТЗ по мере разработки; сохранять гибкость, но фиксировать критичные пункты.

## 3. Дизайн (архитектура продукта)  
**Цель:** выбрать и описать оптимальную архитектуру (DDS – Design Document Specification).  

- **Проблема:** неподходящая архитектура → трудности в реализации и масштабировании.  
- **Решение:** привлекать опытных технических лидов; сравнивать несколько вариантов и документировать выбор.

## 4. Разработка ПО  
**Цель:** реализовать функции согласно DDS, покрыть код unit‑тестами и интеграционными тестами.  

| Проблема | Решение |
|---|---|
| Слабая коммуникация в команде | Daily‑meetings, скрам‑доска, 100 % вовлечённость |
| Невыполнимые сроки | Менеджер аргументирует realistic‑deadlines; приоритетизация задач |
| Добавление неоговоренных фич | Жёстко придерживаться SRS; фиксировать изменения через Change‑Request |

## 5. Тестирование  
**Цель:** выявить и исправить дефекты до выхода продукта, соблюдая критерии качества из SRS.  

- **Проблема:** много времени тратится на поиск причин багов после завершения разработки.  
- **Решение:** тестировать параллельно с разработкой; автоматизировать регрессионные проверки; вовлекать QA уже на ранних этапах.

## 6. Внедрение и поддержка (Deployment + Maintenance)  
**Цель:** выпустить продукт (часто поэтапно), собрать обратную связь, обеспечить эксплуатацию.  

- **Проблема 1:** отсутствие реальной обратной связи от пользователей.  
  - **Решение:** запускать MVP как можно раньше, проводить UAT, приоритизировать дальнейший функционал по отзывам.  
- **Проблема 2:** слабая инфраструктура у клиента (необлачные серверы, частые сбои).  
  - **Решение:** заранее информировать заказчика о рисках, предлагать облачные решения или резервные схемы.
